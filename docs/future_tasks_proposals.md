## 各提案の詳細設計方針

### 1. アプリ内からの自動起動登録・管理
* **背景**: [installer_auto_update_plan.md](file:///C:/Users/dev/src/squirrel-notifier/docs/installer_auto_update_plan.md) の留意事項にある「タスクスケジューラ登録に失敗した場合はログとダイアログで通知し、アプリ内から再登録できる動線を残す」という機能の具体化。
* **設計案**:
  - 設定ウィンドウに「Windows 起動時に自動起動する (タスクスケジューラ登録)」のトグルスイッチを追加。
  - C# の `Microsoft.Win32.TaskScheduler` パッケージ（または `schtasks.exe` コマンドラッパー）を使用し、アプリ内部からタスクの登録/解除/状態取得を直接行う。

### 2. 重複排除キャッシュおよび履歴リストの永続化
* **背景**: 現在、重複排除キャッシュ（`eventId`）および直近のイベント履歴はメモリ内のみで保持されているため、アプリ再起動時に履歴が消失し、同じレビューイベントが再通知される恐れがあります。
* **設計案**:
  - `%LocalAppData%\SquirrelNotifier\cache.json` などのキャッシュ専用ファイルを作成。
  - 通知済み ID（過去100件）と履歴リスト（過去20件）をシリアライズして保存し、アプリ起動時に復元する。
  - **ファイル破損対策**: 書き込み処理中のクラッシュや PC 強制シャットダウンに伴うファイル破損（0バイト化）を防ぐため、一時ファイル（`cache.json.tmp`）に一度書き出し、正常に完了した後に既存ファイルと安全に置き換えるアトミック書き込み（`File.Replace` や `File.Move(..., overwrite: true)` などの API を使用）を実装する。

### 3. エラー状態の可視化とトレイアイコンのUX改善
* **背景**: 現在、MCPサーバーとの通信エラーやコマンドエラーなどの `SubscriptionState.Error` 時に、ウィンドウを開かないとエラーが分かりません。
* **設計案**:
  - エラー状態を表す警告マーク付きのトレイアイコン（例: `squirrel-notifier-error.ico`）を用意。
  - `TrayIconService` に `SetIconState(SubscriptionState state)` メソッドを追加し、エラー時はアイコンを差し替えてユーザーに視覚的に通知。

### 4. MSI インストーラーとパッケージ署名自動化の導入
* **背景**: MSI ベースのインストーラーを用意し、インストールとタスクスケジューラ登録を自動化します。ローカル開発・テスト環境において SmartScreen や信頼されていない発行元による警告を回避・検証するために、自己署名を用いたパッケージングおよび署名フローを構築します。
* **設計案**:
  - WiX Toolset 等を用いた MSI インストーラーの構成。
  - セキュリティ上のリスク（信頼チェーンの破壊や、セキュリティ対策ソフトによるアドウェア/マルウェア的挙動としての誤検知リスク）を回避するため、インストーラーによる「信頼されたルート証明機関」への CA 証明書の自動登録は行いません。
  - 代わりに以下のプロセスを採用します：
    1. テスト用の自己署名証明書を作成し、MSI パッケージまたはアプリ本体に署名を適用。
    2. ローカルの開発・テスト環境において、検証のために必要な証明書のインポート手順（手動インポート手順、または信頼された発行元ストアへの適用など範囲を限定した登録方法）をドキュメント化し、開発者が必要に応じて実行する。

## 実機テスト結果 (2026-06-14 実施)

今後の機能実装に先立ち、現行のビルドおよびインストーラースクリプトを用いた動作検証（実機テスト）を実施しました。

### 1. 検証内容と結果

* **テスト1: `dotnet build` 生成物の起動テスト**
  - **結果**: **失敗 (即時クラッシュ)**
  - **詳細**: `bin/x64/Release/net10.0-windows10.0.26100.0/` 配下の EXE を直接実行、またはタスクスケジューラから起動した場合、イベントログに `0xc000027b`（障害モジュール `Microsoft.UI.Xaml.dll`、終了コード `0xC0000409`）を伴って即時強制終了しました。非パッケージ（Unpackaged）アプリとして動作させる際に必要な Windows App SDK のランタイム DLL や依存アセットが実行ディレクトリに不足していることが原因です。
* **テスト2: `dotnet publish`（自己完結型）生成物の起動テスト**
  - **結果**: **成功 (正常起動・ログ出力確認)**
  - **詳細**: `dotnet publish -c Release -r win-x64 --self-contained true` でパブリッシュした成果物（`publish/` ディレクトリ内）の EXE は、クラッシュすることなく正常に起動し、バックグラウンドでの動作を確認しました。
* **テスト3: アプリケーション動作ログの検証**
  - **結果**: **成功 (エラーハンドリングログの確認)**
  - **詳細**: `%LocalAppData%\SquirrelNotifier\Logs\winui3.log` にログが書き込まれることを確認しました。MCP サーバー未起動による接続エラー (`subscribe-probe failed: fetch failed`) を検知し、アプリがクラッシュせずにエラー状態へと安全に移行していることが確認できました。

## 浮き彫りになった新たな課題と追加タスク（作業予定）

実機テストの結果から、今後のタスクとして以下を解消・実装する必要があります。

### 【課題A】インストーラースクリプトのパス解決の修正
* **現状**: 現在の `install.ps1` は `dotnet build` の出力先（`bin/`）を優先して検索・登録するようになっています。
* **対策**: クラッシュを防ぐため、`install.ps1` ではパブリッシュされた自己完結型成果物（`publish/`）のみを検索・配置対象とするように修正します。また、開発者が誤ってビルド成果物を登録しないよう、警告やチェックを追加します。

### 【課題B】MCP 購読のエラーハンドリングと自動リトライ
* **現状**: MCP サーバー未起動などによる接続エラー発生時、ループが即終了して `Error` 状態で停止します（自動リトライなし）。
* **対策**: 自動更新と同様に、MCP 購読接続にも指数バックオフを用いた自動リトライ処理を導入し、一時的なネットワーク切断やサーバー起動の遅れに対して耐性を持たせます。

## 開発・テストにおけるセキュリティ検証方針

開発およびテスト環境において Windows Defender 等のセキュリティ機能による誤検知（ヒューリスティック検知）を防ぎ、安全な開発を維持するために以下の方針を厳守します。

1. **PowerShell 経由での動的検証の禁止**
   - テストや検証用スクリプトにおいて、`Assembly.LoadFrom`、`Reflection`、`Add-Type`、および Win32 API（`CommandLineToArgvW` や `LocalFree` 等）の `DllImport` を含む動的 DLL ロードやインライン C# コンパイルは一切行いません。
2. **CommandLineToArgvW 等の比較検証方針**
   - 本番コードの `ParseArguments` のロジックと標準 Windows API（`CommandLineToArgvW`）との完全一致検証が必要な場合は、PowerShell 経由ではなく、C# 単体テストプロジェクト（`SquirrelNotifier.WinUI3.Tests`）内に安全な P/Invoke ラッパーを定義して検証を行います。
   - 動的比較が過剰な場合は、代表的なケースを期待値固定としたテーブルテスト（`[Theory]`）形式に留め、OS の API との動的比較は行いません。
3. **セキュリティ除外設定の不採用**
   - セキュリティ警告を回避するために、開発マシンの Defender やセキュリティソフトに除外設定を追加する対応は行いません。
